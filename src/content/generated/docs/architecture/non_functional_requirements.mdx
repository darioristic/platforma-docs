## 1. Multi-Tenancy Requirements

### 1.1 Tenant Isolation Model

| Isolation Level | Implementation | Use Case |
|-----------------|----------------|----------|
| **Logical** | Row-level security, tenant_id column | Standard tenants |
| **Schema** | Separate PostgreSQL schemas | Enterprise tenants |
| **Database** | Dedicated database instance | Regulated industries |
| **Compute** | Dedicated pods/namespaces | Premium SLA |

### 1.2 Data Isolation Requirements

```yaml
data_isolation:
  tenant_context:
    - All database queries MUST include tenant_id filter
    - Tenant context MUST be extracted from JWT token
    - Cross-tenant data access MUST be explicitly denied
    - Superadmin access MUST be audited separately

  storage_isolation:
    - Object storage: Tenant-prefixed paths mandatory
    - Encryption keys: Per-tenant KEK (Key Encryption Key)
    - Backup segregation: Tenant-specific backup policies

  cache_isolation:
    - Redis key prefix: tenant:{tenant_id}:*
    - TTL policies: Per-tenant configurable
    - Memory limits: Per-tenant quotas
```

| Users per tenant | 50 | Unlimited |
| API calls/minute | 100 | 10,000 |
| Storage (GB) | 100 | Custom |

## 2. Audit Logging Requirements

### 2.1 Audit Event Categories

| Category | Retention | Storage |
|----------|-----------|---------|
| Authentication | 2 years | Hot: 90 days, Cold: remainder |
| Authorization | 2 years | Hot: 90 days, Cold: remainder |
| Data Access | 7 years | Hot: 30 days, Cold: remainder |
| Data Modification | 7 years | Hot: 90 days, Cold: remainder |
| System Events | 1 year | Hot: 30 days, Cold: remainder |
| Financial Events | 10 years | Hot: 1 year, Cold: remainder |

### 2.2 Audit Event Schema

```typescript
interface AuditEvent {
  // Identifiers
  eventId: string;           // UUID v7 (time-ordered)
  correlationId: string;     // Request correlation
  traceId: string;           // Distributed trace ID

  // Temporal
  timestamp: string;         // ISO 8601 with microseconds
  timezone: string;          // Source timezone

  // Actor
  actorId: string;
  actorType: 'user' | 'service' | 'system' | 'api_key';
  actorEmail?: string;       // For user actors
  actorIp: string;
  actorUserAgent?: string;

  // Context
  tenantId: string;
  sessionId?: string;
  requestId: string;

  // Action
  action: string;            // e.g., 'user.login', 'order.create'
  actionCategory: string;    // e.g., 'authentication', 'data_modification'
  result: 'success' | 'failure' | 'partial';

  // Resource
  resourceType: string;
  resourceId: string;
  resourceName?: string;

  // Change Data
  previousState?: object;    // For modifications
  newState?: object;         // For modifications
  changedFields?: string[];  // List of modified fields

  // Metadata
  metadata: {
    source: string;          // Service name
    version: string;         // API version
    environment: string;
    datacenter: string;
    [key: string]: unknown;
  };

  // Integrity
  checksum: string;          // SHA-256 of event content
  signature?: string;        // Optional cryptographic signature
}
```

### 2.3 Audit Log Integrity

```yaml
integrity_requirements:
  immutability:
    - Audit logs MUST be append-only
    - No modification or deletion of audit entries
    - Write-once storage for compliance

  chain_of_custody:
    - Each event includes hash of previous event
    - Periodic merkle tree computation
    - External timestamp authority for non-repudiation

  tamper_detection:
    - Hourly integrity verification
    - Alert on any inconsistency
    - Separate audit of audit access

  encryption:
    - At-rest: AES-256-GCM
    - In-transit: TLS 1.3
    - Key rotation: 90 days
```

## 3. Role-Based Access Control (RBAC)

### 3.1 Permission Model

```yaml
permission_structure:
  format: "{domain}:{resource}:{action}"

  examples:
    - "product:catalog:read"
    - "order:*:write"
    - "admin:tenant:manage"
    - "billing:invoice:read"

  actions:
    - create
    - read
    - update
    - delete
    - list
    - execute
    - manage
    - "*" (wildcard)
```

### 3.2 Standard Roles

#### Public Web & Shop

| Role | Permissions |
|------|-------------|
| Anonymous | `product:catalog:read`, `product:pricing:read` |
| Prospect | Anonymous + `cart:*:*`, `order:create:execute` |

#### Customer Portal

| Role | Permissions |
|------|-------------|
| Customer Viewer | `service:own:read`, `billing:own:read`, `order:own:read` |
| Customer Admin | Customer Viewer + `service:own:manage`, `user:tenant:manage` |
| Customer Owner | Customer Admin + `billing:own:manage`, `apikey:own:manage` |

#### Admin Platform

| Role | Permissions |
|------|-------------|
| Support Agent | Read access to customer data, ticket management |
| Operations | Service management, monitoring, alerts |
| Product Manager | Product catalog CRUD, pricing management |
| Billing Admin | Invoice management, payment processing |
| System Admin | Full platform administration |
| Super Admin | All permissions, audit access |

### 3.3 Attribute-Based Access Control (ABAC) Extensions

```typescript
interface AccessPolicy {
  id: string;
  name: string;
  effect: 'allow' | 'deny';

  subjects: {
    roles?: string[];
    users?: string[];
    groups?: string[];
  };

  resources: {
    type: string;
    conditions?: AttributeCondition[];
  };

  actions: string[];

  conditions?: {
    timeRange?: { start: string; end: string };
    ipRange?: string[];
    mfaRequired?: boolean;
    tenantMatch?: boolean;
    customExpression?: string;  // CEL expression
  };
}

interface AttributeCondition {
  attribute: string;
  operator: 'eq' | 'neq' | 'in' | 'contains' | 'startsWith';
  value: unknown;
}
```

### 3.4 Permission Inheritance

```
Organization
    └── Tenant
         ├── inherits: Organization policies
         └── Roles
              ├── inherit: Tenant permissions
              └── Users
                   └── inherit: Role permissions + direct grants
```
