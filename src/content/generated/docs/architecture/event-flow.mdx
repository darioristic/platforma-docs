# Cloud Factory - Service Interconnection Map

**Version:** 3.4 | **Date:** February 17, 2026

This document describes how Cloud Factory platform microservices are interconnected.
All inter-service communication goes **exclusively through Apache Kafka** (async, event-driven).
There are no direct HTTP calls between services — each service is fully autonomous with its own database.

---

## 1. Communication Patterns Overview

The platform uses three communication patterns:

| Pattern               | Technology                     | Description                                                          |
| --------------------- | ------------------------------ | -------------------------------------------------------------------- |
| **Client → Service**  | HTTP via nginx gateway (:8880) | Frontend applications call services exclusively through API Gateway  |
| **Service → Service** | Apache Kafka (:9092)           | Asynchronous communication via domain events                         |
| **Service → Client**  | WebSocket (Socket.IO)          | Real-time notifications from notification-service to browser clients |

There are no synchronous inter-service calls (HTTP, gRPC). Services do not share databases — each has its own PostgreSQL schema or MongoDB collection.

## 6. Notification Service as Event Sink

Notification Service is the central hub that listens to events from all other services
and transforms them into user notifications (in-app and email).

```text
                                    ┌──────────────────────────┐
    order.created ─────────────────►│                          │
    order.fulfilled ───────────────►│                          │
    order.failed ──────────────────►│                          │      ┌──────────────┐
    order.cancelled ───────────────►│                          │ ────►│  Browser      │
                                    │   Notification Service   │      │  (Socket.IO)  │
    provisioning.completed ────────►│   :3006                  │      └──────────────┘
                                    │                          │
    billing.invoice-generated ─────►│   10 event types         │      ┌──────────────┐
    billing.payment-received ──────►│   → in-app notification  │ ────►│  Email        │
                                    │   → email notification   │      │  (SMTP)       │
    support.ticket-created ────────►│                          │      └──────────────┘
    support.comment-added ─────────►│                          │
    support.ticket-resolved ───────►│                          │
                                    └──────────────────────────┘

    For each received event:
    1. Checks user preferences (whether they want in-app/email for that type)
    2. Creates a Notification entity in the database
    3. Sends via Socket.IO WebSocket if the user is online
    4. Sends email via SMTP if the user has enabled it
    5. Publishes notification.delivered or notification.failed
```

## 8. Kafka Consumer Groups

Each service has one consumer group. Kafka guarantees that within a group
each partition goes to exactly one consumer — this enables horizontal scaling.

| Consumer Group                                 | Service      | Subscribed Topics |
| ---------------------------------------------- | ------------ | ----------------- |
| `order-service-provisioning-events-consumer`   | Order        | 6 topics          |
| `provisioning-service-order-events-consumer`   | Provisioning | 2 topics          |
| `billing-service-provisioning-events-consumer` | Billing      | 4 topics          |
| `notification-service-order-events-consumer`   | Notification | 10 topics         |
| `dlq-handler-all-consumer`                     | DLQ Handler  | all (dead letter) |

## 9. Dead Letter Queue (DLQ)

When a consumer fails to process a message after all retry attempts,
the message is sent to a Dead Letter Queue for later analysis.

```text
  ┌─────────────┐     retry (max 3x)      ┌─────────────┐
  │   Kafka     │ ────────────────────────►│  Consumer   │
  │   Topic     │                          │  Handler    │
  └─────────────┘                          └──────┬──────┘
                                                  │
                                           fail after retries
                                                  │
                                                  ▼
                                           ┌─────────────┐
                                           │    DLQ       │
                                           │   Topic      │
                                           └──────┬──────┘
                                                  │
                                                  ▼
                                           ┌─────────────┐
                                           │ DLQ Handler  │
                                           │ (log, alert) │
                                           └─────────────┘

  Retry configuration:
  ┌──────────────┬─────────┬──────────┬──────────┬────────────┐
  │ Profile      │ Retries │ Initial  │ Max      │ Multiplier │
  ├──────────────┼─────────┼──────────┼──────────┼────────────┤
  │ Default      │ 3       │ 1s       │ 30s      │ 2x         │
  │ Critical     │ 5       │ 500ms    │ 60s      │ 2x         │
  │ Non-critical │ 2       │ 2s       │ 10s      │ 2x         │
  └──────────────┴─────────┴──────────┴──────────┴────────────┘
```

## 10. Data Isolation
