# Cloud Factory Platform - Architecture Overview

## Enterprise Cloud Business Platform for Telecommunications Operator

**Document Version:** 1.0.0
**Classification:** Internal - Architecture

---

## 1. System Overview

Cloud Factory is an enterprise cloud business platform consisting of three strictly separated systems:

| System | Purpose | Users |
|--------|---------|-------|
| **Public Web & Shop** | Marketing, catalog, service ordering | Anonymous users, prospects |
| **Customer Portal** | Self-service, billing, support | Authenticated customers |
| **Admin Platform** | Operations, administration, integrations | Internal telco staff |

### 1.1 Architectural Principles

1. **Strict System Separation** - Each system is an independent deployable unit
2. **API-First Communication** - Systems communicate only via versioned APIs and events
3. **Domain-Driven Design** - Clear bounded contexts with explicit contracts
4. **Event Sourcing for Audit** - Critical domains maintain full audit trail
5. **Multi-Tenancy by Design** - Tenant isolation at data and compute levels
6. **Zero-Trust Security** - All inter-service communication authenticated and authorized

## 5. Security Architecture

### 5.1 Authentication Flows

| System | Method | Provider |
|--------|--------|----------|
| Public Web | Anonymous + OAuth2 (ordering) | Keycloak |
| Customer Portal | OAuth2 + OIDC | Keycloak |
| Admin Platform | OAuth2 + OIDC + MFA | Keycloak + Corp IdP |

### 5.2 Authorization Model

- **RBAC** for coarse-grained permissions
- **ABAC** for fine-grained resource access
- **Tenant-scoped** policies for multi-tenancy

### 5.3 Data Protection

| Data Class | At Rest | In Transit |
|------------|---------|------------|
| PII | AES-256 (field-level) | TLS 1.3 |
| Financial | AES-256 + HSM | mTLS |
| Audit Logs | Immutable + Signed | mTLS |

## 6. Integration Patterns

### 6.1 Synchronous (REST)

- Request/Response for queries
- Circuit breaker pattern (Resilience4j)
- Retry with exponential backoff
- Timeout policies per endpoint

### 6.2 Asynchronous (Events)

- Domain events via Kafka
- Exactly-once semantics for critical flows
- Dead letter queues for failed processing
- Event schema registry (Confluent)

### 6.3 Telco BSS/OSS Integration

| System | Protocol | Pattern |
|--------|----------|---------|
| CRM | REST/SOAP | Adapter |
| Billing | TM Forum APIs | Direct |
| Inventory | REST | Adapter |
| Network/Provisioning | NETCONF/YANG | Gateway |

## 7. Scalability Design

### 7.1 Horizontal Scaling Triggers

| Metric | Threshold | Action |
|--------|-----------|--------|
| CPU | > 70% sustained | Scale out |
| Memory | > 80% | Scale out |
| Request latency p99 | > 500ms | Scale out |
| Queue depth | > 1000 | Scale consumers |

### 7.2 Database Scaling Strategy

- **Read replicas** for query-heavy workloads
- **Sharding** by tenant_id for large tenants
- **Connection pooling** via PgBouncer
- **Caching layer** for hot data

## 8. Failure Isolation

### 8.1 Bulkhead Pattern

Each system operates in isolated failure domains:

```
┌─────────────────────────────────────────────────┐
│              FAILURE DOMAIN: PUBLIC WEB         │
│  ┌─────────┐ ┌─────────┐ ┌─────────────────┐   │
│  │ Web App │ │  BFF    │ │ Dedicated Cache │   │
│  └─────────┘ └─────────┘ └─────────────────┘   │
└─────────────────────────────────────────────────┘
         │ (Circuit Breaker)
         ▼
┌─────────────────────────────────────────────────┐
│          SHARED SERVICES (Degradable)           │
└─────────────────────────────────────────────────┘
```

### 8.2 Graceful Degradation

| Failure | Degradation Strategy |
|---------|---------------------|
| Product Service down | Serve from cache |
| Billing integration down | Queue for retry |
| Search unavailable | Fallback to DB query |

## 10. Service Endpoints & Ports

### 10.1 Local Development Ports

| Service | Port | Base URL |
|---------|------|----------|
| Product Service | 3001 | `http://localhost:3001` |
| Order Service | 3002 | `http://localhost:3002` |

| Public Web (Next.js) | 3000 | `http://localhost:3000` |
| Admin Platform (Vite) | 3100 | `http://localhost:3100` |
| Customer Portal (Vite) | 3200 | `http://localhost:3200` |

### 10.2 Health Check Endpoints

All backend services expose standardized health check endpoints via `@nestjs/terminus`:

| Endpoint | Purpose | Auth Required |
|----------|---------|---------------|
| `GET /health` | Liveness probe - confirms service is running | No |
| `GET /ready` | Readiness probe - confirms dependencies are available | No |

Readiness checks per service:

| Service | Database Check | Kafka Check |
|---------|---------------|-------------|
| Identity Service | TypeORM (PostgreSQL) | - |
| Product Service | Mongoose (MongoDB) | - |
| Order Service | TypeORM (PostgreSQL) | KafkaProducerService |
| Provisioning Service | TypeORM (PostgreSQL) | KafkaProducerService |
| Billing Integration | TypeORM (PostgreSQL) | KafkaProducerService |

### 10.3 CORS & Security Headers

All services are configured with:

- **Helmet** - HTTP security headers (X-Content-Type-Options, X-Frame-Options, CSP, etc.)
- **CORS** - Configurable via `CORS_ORIGIN` environment variable (comma-separated origins)

```
CORS_ORIGIN=http://localhost:3000,http://localhost:3100,http://localhost:3200
```

CORS settings:
- Methods: `GET, POST, PUT, PATCH, DELETE, OPTIONS`
- Allowed headers: `Content-Type, Authorization`
- Credentials: enabled

### 10.4 Environment Validation

All services validate environment variables at startup using **Zod** schemas (`src/config/env.validation.ts`). Invalid configuration causes immediate startup failure with descriptive error messages.

| Service | Key Validated Variables |
|---------|----------------------|
| Identity | `DB_*`, `JWT_SECRET`, `JWT_EXPIRES_IN`, `JWT_REFRESH_SECRET`, `CORS_ORIGIN` |
| Product | `MONGODB_URI`, `CORS_ORIGIN` |
| Order | `DB_*`, `KAFKA_BROKERS`, `CORS_ORIGIN` |
| Provisioning | `DB_*`, `KAFKA_BROKERS`, `CORS_ORIGIN` |
| Billing | `DB_*`, `KAFKA_BROKERS`, `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `CORS_ORIGIN` |

All variables have sensible `.default()` values for local development.

### 10.5 API Key Management

The Identity Service provides API key management for programmatic access:

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api-keys` | GET | List API keys (optional `?userId` filter) |
| `/api-keys` | POST | Create new API key (returns raw key once) |
| `/api-keys/:id` | DELETE | Revoke an API key |

Keys are stored as SHA-256 hashes with a prefix (`cf_prod_` or `cf_dev_`).
